## Зачем нужны расширения XDTO-пакетов

EnterpriseData задаёт **типовой состав** свойств для каждого объекта (Документ.Реализация имеет фиксированный набор полей). При интеграции с типовыми конфигурациями (1С:ERP, УТ, БП) нельзя:

- Снять конфигурацию с поддержки — потеряешь автообновления
- Менять стандартные XDTO-пакеты — это вендорские артефакты
- Удалять/переименовывать поля в стандартном составе — нарушит обмен с другими типовыми

**Решение 1С:** механизм **расширений XDTO-пакетов** (extension packages):
- Создаётся **отдельный** XDTO-пакет в другом namespace (например `http://v8.1c.ru/edi/edi_ext_example/EnterpriseData/1.11` вместо стандартного `http://v8.1c.ru/edi/edi_stnd/EnterpriseData/1.11`)
- Этот пакет **импортирует** стандартный пакет
- В нём объявляются объекты с теми же именами (`Справочник.Номенклатура`, `КлючевыеСвойстваНоменклатура`), но содержащие **только новые свойства**
- В XML-сообщении новые свойства появляются в виде `<ext1:МойКодКлассификации>...</ext1:МойКодКлассификации>`

Это аналогично концепции `xs:extension` в XSD, но на уровне namespaces, а не наследования типов.

## Структура решения

Чтобы добавить в стандартный обмен поле `МойКодКлассификации` для `Справочник.Номенклатура`:

### Шаг 1. Расширение конфигурации 1С

Создаётся расширение конфигурации (cfe), в котором справочнику `Номенклатура` добавляется реквизит `МойКодКлассификации` (String, например).

### Шаг 2. XDTO-пакет-расширение

В расширении создаётся XDTO-пакет с namespace `http://v8.1c.ru/edi/edi_ext_example/EnterpriseData/1.11`. В нём:

- **Директива импорта** на стандартный пакет: `http://v8.1c.ru/edi/edi_stnd/EnterpriseData/1.11`
- Также импорт `http://www.1c.ru/SSL/Exchange/Message` (базовая wrapper-схема, см. `ExchangeMessage.xsd`)
- **Объекты расширения** — копии стандартных по именам, но с **только новыми свойствами**:
  - `Справочник.Номенклатура` → новое свойство `ЕдиницаИзмеренияОтчетов` (тип `КлючевыеСвойстваЕдиницаИзмерения`)
  - `КлючевыеСвойстваНоменклатура` → новое свойство `МойКодКлассификации` (тип `xs:string`)
  - `КлючевыеСвойстваЕдиницаИзмерения` (обязательно забрать, т.к. используется в типе нового свойства)

Важно: **порядок добавляемых свойств должен совпадать в источнике и приёмнике** (по URI). Иначе при загрузке будет ошибка чтения.

### Шаг 3. Регистрация расширения в БСП

В общем модуле `ОбменДаннымиПереопределяемый` поправить процедуру `ПриПолученииДоступныхРасширенийФормата`:

```bsl
Процедура ПриПолученииДоступныхРасширенийФормата(РасширенияФормата)
    РасширенияФормата.Вставить(
        "http://v8.1c.ru/edi/edi_ext_example/EnterpriseData/1.11", "1.0"
    );
КонецПроцедуры
```

### Шаг 4. BSL правила выгрузки с расширением

В модуле `МенеджерОбменаЧерезУниверсальныйФормат` для каждого расширяемого ПКО:

```bsl
&Вместо("ДобавитьПКО_Справочник_Номенклатура_Отправка")
Процедура ДемоРасширений_ДобавитьПКО_Справочник_Номенклатура_Отправка(ПравилаКонвертации)
    ПравилоКонвертации = ОбменДаннымиXDTOСервер.ИнициализироватьПравилоКонвертацииОбъекта(...);
    // ... стандартные ДобавитьПКС ...
    
    // === Расширение XDTO-пакета ===
    ПространствоИмен = "http://v8.1c.ru/edi/edi_ext_example/EnterpriseData/1.11";
    ОбменДаннымиXDTOСервер.ИнициализироватьРасширениеПравилаКонвертацииОбъекта(
        ПравилоКонвертации, ПространствоИмен
    );
    
    ДемоРасширений_ДобавитьПКС(
        СвойстваШапки,
        "ДемоРасширений_МойКодКлассификации",
        "МойКодКлассификации",  // имя в XDTO
        -1, ,
        ПространствоИмен        // ← КЛЮЧЕВОЙ параметр
    );
КонецПроцедуры
```

### Шаг 5. Обработчик `ПриОтправкеДанных` — заполнение

```bsl
// Внутри обработчика, добавляем расширенное свойство в XDTO
Если НЕ ПустаяСтрока(ДанныеИБ.ДемоРасширений_МойКодКлассификации) Тогда
    ДанныеXDTO.КлючевыеСвойства.Вставить(
        "МойКодКлассификации",
        ДанныеИБ.ДемоРасширений_МойКодКлассификации
    );
КонецЕсли;
```

## Что это значит для архитектуры Konvey

### Модель данных (Sprint 1+ ready, Sprint 2 реализация)

`XsdComplexType` и `XsdField` должны иметь поле:

```python
class XsdField(BaseModel):
    name: str
    type_ref: str
    namespace: str | None = None  # None = базовый namespace; "http://...ext_example/..." = расширение
    is_extension: bool = False
    ...
```

`FieldMapping`:

```python
class FieldMapping(BaseModel):
    ed_field: str
    ed_namespace: str | None = None  # для расширений
    extension_namespace_id: str | None = None  # ID известного расширения, чтобы группировать в BSL
    ...
```

### Парсер XSD (Sprint 1)

`xsd_parser.parse_xsd()` должен:

1. Читать `targetNamespace` корня
2. Различать "стандартный" пакет (namespace начинается с `edi_stnd/`) и пакет-расширение (другой namespace)
3. Парсить `xs:import` — собирать **граф зависимостей пакетов**
4. Объединять типы из всех импортированных пакетов в единый namespace-aware каталог
5. Возвращать `EnterpriseDataSchema` со списком namespaces:

```python
class EnterpriseDataSchema(BaseModel):
    primary_namespace: str  # тот что был передан как корневой XSD
    namespaces: list[str]   # все известные (импортированные)
    extension_namespaces: list[str]  # подмножество, относящееся к расширениям
    simple_types: dict[str, XsdSimpleType]
    complex_types: dict[str, XsdComplexType]  # key = "<namespace>:<name>"
```

### UI (Sprint 2)

В Workspace для типов EnterpriseData:

- Базовые поля отображаются **обычным цветом**
- Поля расширений — **с префиксом `ext1:` и иконкой** (например, маленький значок «✦»)
- При маппинге на расширенное поле — Inspector показывает `Extension namespace: http://...ext_example/...`
- Кнопка «Скрыть расширения» в фильтрах (если пользователь работает только с типовым обменом)

### BSL генератор (Sprint 3)

При генерации `МенеджерОбменаЧерезУниверсальныйФормат`:

1. Группировать FieldMappings по `extension_namespace_id`
2. Для каждого расширения вставлять блок:
   ```bsl
   ПространствоИмен = "<URI>";
   ОбменДаннымиXDTOСервер.ИнициализироватьРасширениеПравилаКонвертацииОбъекта(
       ПравилоКонвертации, ПространствоИмен);
   <prefix>_ДобавитьПКС(СвойстваШапки, ..., ПространствоИмен);
   ```
3. В обработчике `ПриОтправкеДанных` добавлять условные блоки заполнения для каждого расширенного свойства
4. Также генерировать или модифицировать `ОбменДаннымиПереопределяемый.ПриПолученииДоступныхРасширенийФормата`

### Validation (Sprint 2)

При валидации правил:

- Проверить что для каждого `is_extension=True` FieldMapping существует `xs:import` базового пакета в XSD расширения
- Проверить что namespaces в FieldMappings зарегистрированы в `Project.extension_namespaces`

## Связь с ExchangeMessage.xsd

Файл `examples/enterprise_data_docs/ExchangeMessage.xsd` определяет:

```xml
<xs:schema targetNamespace="http://www.1c.ru/SSL/Exchange/Message">
  <xs:complexType name="Confirmation">
    <xs:sequence>
      <xs:element name="ExchangePlan" type="xs:string"/>
      <xs:element name="To" type="xs:string"/>
      <xs:element name="From" type="xs:string"/>
      <xs:element name="MessageNo" type="xs:integer"/>
      <xs:element name="ReceivedNo" type="xs:integer"/>
    </xs:sequence>
  </xs:complexType>
  
  <xs:complexType name="Header">
    <xs:sequence>
      <xs:element name="Format" type="xs:string"/>
      <xs:element name="CreationDate" type="xs:dateTime"/>
      <xs:element name="Confirmation" type="tns:Confirmation" minOccurs="0"/>
      <xs:element name="AvailableVersion" type="xs:string" maxOccurs="100"/>
    </xs:sequence>
  </xs:complexType>
  
  <xs:complexType name="Object" abstract="true">
    <xs:sequence>
      <xs:element name="AdditionalInfo" minOccurs="0"/>
    </xs:sequence>
  </xs:complexType>
  
  <xs:simpleType name="Ref">
    <xs:restriction base="xs:string"/>
  </xs:simpleType>
</xs:schema>
```

**Все объекты EnterpriseData наследуются от `Object`** через `xs:extension base="ns_msg:Object"`. Это объясняет, почему в категоризации видим `parent_type = "Object"` у `Справочник.X` / `Документ.X`.

Когда Konvey будет генерировать **sample XML** (Sprint 1.5+), нужно использовать структуру:

```xml
<КонвертацияФормата>
  <Header>
    <Format>http://v8.1c.ru/edi/edi_stnd/EnterpriseData/1.8</Format>
    <CreationDate>2026-05-16T22:30:00</CreationDate>
    <AvailableVersion>1.8</AvailableVersion>
  </Header>
  <Body>
    <Справочник.Контрагенты>...</Справочник.Контрагенты>
    <Документ.РеализацияТоваровУслуг>...</Документ.РеализацияТоваровУслуг>
  </Body>
</КонвертацияФормата>
```

## TL;DR для Sprint 1

- **Не реализовывать `ext1:*`** в Sprint 1 — слишком много работы
- **НО** заложить в моделях `XsdField.namespace: str | None` и `FieldMapping.ed_namespace: str | None`
- Парсер XSD читает только корневой XSD (без import) в Sprint 1; multi-namespace — Sprint 2
- В UI расширения не отображать (но если в маппинге появится `ed_namespace != None` — показать предупреждение «Sprint 2 feature»)
- BSL генератор — Sprint 3 — будет генерировать только базовые ПКС, без расширений
